# 提案

## 问题

<!-- 用 2-5 条要点描述用户问题或业务问题。 -->

- `quick-sdd` 当前的 `pm / ra / ta / dev / qa` 角色技能主要围绕 SDD 流程编排，缺少各自岗位应具备的专业实践，例如需求分析方法、架构权衡、工程实现准则、测试与验收策略。
- `D:/code/github/claude` 下已经沉淀了多个 agent 项目与角色体系，但这些能力分散在不同仓库、不同目录层级与不同语言版本中，尚未被系统盘点和归纳。
- 如果直接复制现成 prompt，而不先做角色映射、重复项去重和能力分层，`quick-sdd` 很容易失去“轻量、可维护、可安装”的特点。
- 目前 `quick-sdd` 仓库自身还没有承载这项工作的 `codespec/`，不利于后续按阶段推进、追踪和验证。

## 目标

<!-- 描述期望结果，而不是实现方式。 -->

- 系统性盘点 `D:/code/github/claude` 下所有 agent 项目，建立可追踪的项目清单、角色索引和去重规则。
- 提炼可复用的角色优秀实践，并沉淀成面向 `quick-sdd` 的 `PM / RA / TA / DEV / QA` 能力矩阵。
- 在不破坏 Quick SDD 轻量流程边界的前提下，把专业能力以可维护的方式注入各角色 skill、agent 配置与参考资料。
- 为后续安装、使用和验证保留清晰文档与可恢复状态，让这项工作能够按 task 持续推进到完成。

## 范围内

<!-- 列出这个 feature 在 v1 中会包含的内容。 -->

- 盘点 `claude` 目录下的顶层 agent 项目，包括 `agents`、`everything-claude-code`、`everything-claude-code-zh`、`harmony-claude-code`、`Claude-Code-Game-Studios`、`edict`。
- 建立研究产物，包括项目清单、角色能力矩阵、赋能设计说明以及和 `quick-sdd` 文件结构的映射。
- 更新 `quick-sdd` 中与角色赋能直接相关的文件，例如各角色的 `SKILL.md`、`agents/openai.yaml`、必要的 `references/` 文档以及面向安装/使用者的说明文档。
- 保留并强化 Quick SDD 的角色边界，让新增能力服务于 `pm -> ra -> ta -> dev -> qa` 主流程，而不是替代流程本身。
- 为这项改造准备最小验证闭环，包括任务完成标准、文档一致性检查和验收记录。

## 范围外

<!-- 列出容易混入但不应纳入本 feature 的相邻工作。 -->

- 不把 `claude` 仓库中的全部 agent 或 skill 原样搬运进 `quick-sdd`。
- 不重构 Quick SDD 的整体流程框架、状态机或安装系统，除非这项赋能工作直接需要且有明确证据支持。
- 不尝试一次性覆盖所有垂直领域专家能力；本轮只聚焦 `quick-sdd` 五个核心角色的通用专业能力。
- 不新增与当前目标无关的 UI、演示工程或独立自动化系统。

## 风险

<!-- 记录交付、产品、技术或依赖风险。 -->

- `claude` 目录规模较大，存在英文版、中文版和变体项目，若没有去重策略，研究工作容易重复。
- `quick-sdd` 角色与外部项目中的角色并非一一对应，映射时需要避免“硬套角色名”造成职责漂移。
- 如果新增实践过多或过长，角色 skill 会变得臃肿，反而削弱 Quick SDD 的轻量优势。
- 研究产物若缺少来源追踪，后续维护时难以判断某条实践来自哪个项目、是否仍值得保留。

## 待确认问题

<!-- 可选。如果没有待确认事项，可以删除这一节。 -->

- `everything-claude-code-zh` 与 `harmony-claude-code` 中哪些内容视为主源、哪些内容仅作为本地化补充，需要在研究阶段显式标记。
- 是否需要把部分优秀实践沉淀为共享参考文档，而不是重复写入每个角色 `SKILL.md`，需要在设计阶段做结构性决策。
